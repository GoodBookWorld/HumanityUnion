# DOMAIN_LANGUAGE

## Capability 02 — Participation

### Epic 02 — Collaborative Analysis

Version 1.0

---

# Purpose

This document defines the official ubiquitous language for Collaborative Analysis.

All architecture, implementation, APIs, UI, documentation and discussions must use these terms consistently.

No synonyms should replace approved domain terminology.

---

# Mission

A shared language enables a shared understanding.

Collaborative Analysis is built around structured participation rather than free-form discussion.

---

# Core Domain Terms

## Initiative

A proposal created by a participant that aims to solve a problem or improve a situation.

An Initiative owns exactly one Collaborative Analysis.

---

## Collaborative Analysis

The structured process through which a community evaluates and improves an Initiative before formal decision-making.

Collaborative Analysis is not a discussion forum.

---

## Contribution

A structured analytical input submitted by a participant.

Every Contribution has a clear purpose.

Examples include:

- Evidence
- Question
- Alternative
- Reference
- Expert Opinion
- Clarification
- Correction
- Summary Proposal

A Contribution is the primary unit of participation.

---

## Signal

A lightweight structured reaction that communicates analytical feedback without requiring a full Contribution.

Signals complement Contributions.

Signals do not replace reasoning.

---

## Analysis Summary

A structured overview of the current state of Collaborative Analysis.

It summarizes community understanding without replacing individual Contributions.

Analysis Summary represents the current analytical picture.

---

## Progress Policy

A configurable set of requirements that determines when an Initiative is ready to advance.

Examples include:

- minimum Contributions;
- required Signals;
- expert review;
- regional review;
- participation thresholds.

Progress Policy belongs to the Initiative.

---

## Readiness

A measurable indicator showing how close an Initiative is to meeting its Progress Policy.

Readiness is transparent.

Participants always understand:

- current progress;
- remaining requirements;
- missing analytical work.

---

## Analysis Metrics

Quantitative indicators describing Collaborative Analysis activity.

Examples include:

- Contributions;
- Signals;
- participant diversity;
- evidence count;
- expert participation.

Metrics describe analysis.

They do not determine decisions.

---

# Contribution Types

The following Contribution types are approved:

- Evidence
- Question
- Alternative
- Clarification
- Reference
- Expert Opinion
- Summary Proposal
- Correction

Future Contribution types require architectural review.

# Signal Types

Signals are intentionally lightweight.

Examples include:

- Needs Clarification
- Strong Evidence
- Weak Evidence
- Duplicate
- Needs Expert Review
- Regional Impact
- High Priority
- Ready for Poll

The complete taxonomy is defined separately.

# Participant Roles

## Initiative Steward

Creates and maintains the Initiative.

Defines the Progress Policy.

Coordinates Initiative evolution.

---

## Contributor

Provides Contributions.

May also submit Signals.

---

## Reviewer

Evaluates Contributions according to approved policies.

Reviewer responsibilities are defined in future Epics.

---

## Expert

Provides recognized domain expertise.

Expert validation remains transparent.

---

## Collaborative Analysis

Structured improvement stage.

---

Initiative is prepared for community decision-making.

---

# Prohibited Terminology

The following terms should not be used within this Epic:

- Discussion
- Comment
- Like
- Dislike
- Thread
- Forum
- Chat
- Flame War
- Reply Chain

Preferred terminology:

| Avoid | Use |
|--------|-----|
| Discussion | Collaborative Analysis |
| Comment | Contribution |
| Like | Signal |
| Thread | Analysis Session (if needed) |
| Discussion Summary | Analysis Summary |

# Final Principle

Language shapes architecture.

Architecture shapes implementation.

Collaborative Analysis must always be described using language that reflects structured collective reasoning rather than unstructured discussion.
