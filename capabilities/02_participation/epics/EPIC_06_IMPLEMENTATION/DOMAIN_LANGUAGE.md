# Purpose

This document defines the ubiquitous language for the **Implementation** domain.

Its purpose is to establish one consistent vocabulary for architects, engineers, designers and future contributors working on Epic 06.

Implementation is **Stage 7** of the Humanity Union Participation Pipeline.

It follows eligible prior stages — including **Implementation Commitment** — and answers the civic question:

**"What is being done?"**

Every architectural document, implementation guide and code artifact for Epic 06 must use the terminology defined here.

This document defines language only.

It does not define implementation.

Terminology must conform to:

- `project/architecture/governance/PARTICIPATION_ARCHITECTURE_FREEZE.md`
- `capabilities/02_participation/epics/EPIC_05_IMPLEMENTATION_COMMITMENT/EPIC_05_ARCHITECTURE_FREEZE.md`

# Core Terms

## Implementation

**Implementation** is the civic process through which a community records collective progress toward executing an approved direction.

Implementation:

- follows eligible prior pipeline stages by reference;
- records milestones, achievements and evidence of collective action;
- derives collective progress and completion assessment;
- preserves historical integrity of what was done;
- publishes operational and public transparency during execution.

Implementation is not project management.

It is not employment administration.

It is the execution record stage of the Participation Pipeline.

## Implementation Phase

An **Implementation Phase** is a major segment of the Implementation lifecycle representing a coherent period of collective execution toward approved direction.

An Implementation Phase:

- belongs to one Implementation aggregate context;
- groups related milestones and achievements;
- supports orientation without replacing milestone meaning;
- follows policy-defined phase structure where applicable.

Phases organize collective execution narrative.

They do not assign personal work.

## Milestone

A **Milestone** is a defined checkpoint in the Implementation lifecycle representing an expected collective outcome or stage of progress toward completion.

A Milestone:

- expresses civic meaning — not a personal to-do;
- may be required or optional under Completion Criteria;
- is evaluated through recorded Achievements and Evidence;
- remains part of the historical record once reached or assessed.

Milestones describe what the community intended to accomplish — not who must perform labor.

## Achievement

An **Achievement** is a recorded collective accomplishment that satisfies or advances progress toward a Milestone.

An Achievement:

- records what was accomplished in civic terms;
- may reference supporting Evidence;
- contributes to derived Collective Progress;
- is recorded through accountable Implementation Update flows;
- preserves historical integrity once recorded under policy.

Achievement records collective accomplishment.

It is not a task assignment or personal work log entry.

## Collective Progress

**Collective Progress** is a **derived value** representing how far collective implementation has advanced toward defined Milestones and Completion Criteria.

Collective Progress:

- is computed from recorded Achievements, Milestone state and applicable policy;
- reflects community-level advancement — not individual productivity metrics;
- is never manually edited as a substitute for recorded truth;
- may be presented through Progress Indicators and Progress Snapshots.

Collective Progress describes what the community has accomplished together.

It does not rank participants.

## Completion

**Completion** is a **derived value** representing whether Implementation has satisfied applicable Completion Criteria and reached a closed civic execution state.

Completion:

- is evaluated — never manually toggled as authority;
- reflects recorded achievements against required milestones and policy;
- does not re-approve the underlying Collective Decision;
- preserves auditable meaning of why implementation is considered complete.

Completion means the civic execution record is fulfilled — not that every possible future action has ended everywhere in society.

## Completion Assessment

A **Completion Assessment** is the derived evaluation process that determines whether recorded collective progress satisfies Completion Criteria.

Completion Assessment:

- compares Achievements, Milestone satisfaction and policy rules;
- produces Completion and Completion Indicator outcomes;
- remains distinct from Collective Decision approval or Implementation Commitment readiness;
- preserves transparency about satisfied and unsatisfied criteria.

Assessment evaluates recorded truth.

It does not substitute for human judgment about civic merit outside defined criteria.

## Progress Snapshot

A **Progress Snapshot** is a point-in-time derived representation of Collective Progress suitable for workspace or public display.

A Progress Snapshot:

- captures aggregate progress state at a defined moment;
- is derived — not manually authored as source of truth;
- may include Progress Indicators and milestone satisfaction summaries;
- supports transparency without exposing private operational detail beyond policy.

Snapshots communicate state.

They do not mutate implementation truth.

## Implementation Status

**Implementation Status** is the lifecycle state of the Implementation aggregate within its defined state machine.

Implementation Status:

- belongs to the Implementation domain;
- must not reuse Petition, Collective Decision or Implementation Commitment statuses as synonyms;
- governs which commands and experience surfaces are permitted;
- is distinct from derived Progress Indicators or Completion Indicators.

Status describes aggregate lifecycle — not derived progress percentage alone.

## Implementation Timeline

An **Implementation Timeline** is the ordered presentation of Implementation Phases, Milestones, Achievements and significant Implementation Updates over time.

An Implementation Timeline:

- supports civic orientation and public transparency;
- presents collective execution history in comprehensible sequence;
- is informational — not a scheduling or assignment system in Version 1;
- may reference Evidence Links where policy permits.

The timeline narrates what happened.

It does not manage calendars or deadlines for individuals.

## Progress Indicator

A **Progress Indicator** is a derived presentation signal expressing Collective Progress toward Milestones or Completion Criteria.

Progress Indicators:

- are computed from recorded achievements and milestone state;
- use calm, factual civic language;
- must not use gamification, ranking or urgency framing;
- are read-only at experience surfaces unless underlying recorded truth changes.

Indicators summarize derived progress.

They are not editable progress fields.

## Completion Indicator

A **Completion Indicator** is a derived presentation signal expressing progress toward or attainment of Completion.

Completion Indicators:

- reflect Completion Assessment outcomes;
- distinguish required milestone satisfaction from optional milestone context where applicable;
- must not imply manual approval authority;
- remain subordinate to explicit Completion Criteria definitions.

Indicators communicate assessment results.

They do not authorize bypass of governance.

## Evidence

**Evidence** is factual supporting material associated with an Achievement or Implementation Update demonstrating that a collective accomplishment occurred or progressed.

Evidence:

- supports transparency and auditability;
- may include references, attachments or links under policy;
- is distinct from opinion, commentary or persuasive narrative;
- preserves historical meaning once recorded under applicable immutability rules.

Evidence substantiates accomplishment.

It does not replace Achievement recording.

## Evidence Reference

An **Evidence Reference** is an identifier or approved citation pointing to Evidence without necessarily embedding full content in the aggregate.

Evidence References:

- maintain aggregate independence;
- may point to external documents, records or approved repositories;
- support verification without duplicating external system ownership.

References connect truth.

They do not merge external domains into Implementation ownership.

## Evidence Attachment

An **Evidence Attachment** is Evidence content stored or linked as part of an Implementation record under platform policy.

Evidence Attachments:

- remain subordinate to Achievement or Implementation Update records;
- follow visibility rules for operational and public surfaces;
- must not expose private participant detail beyond approved policy.

Attachments support substantiation.

They are not general file storage for unrelated collaboration.

## Evidence Link

An **Evidence Link** is a URL or platform-resolved link connecting an Achievement or Implementation Update to external or internal Evidence.

Evidence Links:

- provide access to supporting material;
- remain references — not ownership claims over linked systems;
- must be stable enough for civic audit where policy requires.

Links increase transparency.

They do not integrate external tools into Version 1 scope.

## Implementation Update

An **Implementation Update** is an accountable recorded statement of collective implementation progress, context or accomplishment change within the Implementation aggregate.

An Implementation Update:

- is recorded through canonical operational commands;
- may reference Milestones, Achievements or Evidence;
- contributes to derived Collective Progress;
- preserves historical integrity once recorded under policy.

Updates record civic execution narrative.

They are not chat messages or informal status posts.

## Progress Report

A **Progress Report** is a structured presentation of Collective Progress derived from Implementation Updates, Achievements and Milestone state.

A Progress Report:

- summarizes collective advancement in civic language;
- may appear in operational workspace or public projection forms;
- is derived and informational;
- must not expose individual work tracking beyond approved aggregate rules.

Reports communicate collective state.

They are not performance reviews of participants.

## Community Observer

A **Community Observer** is any person — registered or public — who reads Implementation visibility surfaces without holding an accountable recording role for collective achievements.

Community Observers:

- may read Public Implementation projections;
- may share approved public references;
- do not record Achievements or Implementation Updates unless policy assigns them an accountable operational role;
- benefit from transparency without operational control by default.

Observation is permitted.

Accountable recording requires operational participation where policy defines it.

## Implementation Workspace

The **Implementation Workspace** is the operational environment where registered participants orient within Implementation, review collective progress and record accountable Implementation Updates and Achievements where authorized.

The Implementation Workspace:

- answers the five standard participant orientation questions defined in the Participation Architecture Freeze;
- presents derived progress and completion context before primary action;
- separates operational truth from Public Implementation;
- does not function as a task board, calendar or messaging system in Version 1.

The workspace supports collective execution transparency.

It does not replace external coordination tools.

## Public Implementation

**Public Implementation** is the read-only public projection surface presenting collective implementation progress and completion context to society, observers and Community Observers.

Public Implementation:

- exposes aggregate civic meaning approved for public display;
- includes derived progress and completion indicators where policy permits;
- includes share reference and participation entry guidance where applicable;
- must not expose private operational internals or individual work logs by default.

Public Implementation supports transparency.

It is not the accountable recording surface.

## Implementation Visibility

**Implementation Visibility** is the policy-governed boundary defining what Implementation information appears on operational versus public surfaces.

Implementation Visibility:

- separates operational detail from public aggregate presentation;
- protects participant dignity and privacy by default;
- increases transparency through collective indicators — not exposure of private labor tracking;
- follows Explicit Publicity platform principles.

Visibility rules are policy-defined.

They must not be bypassed for convenience.

## Completion Criteria

**Completion Criteria** are the policy-defined conditions that recorded collective progress must satisfy for Completion to be derived.

Completion Criteria may include:

- required milestone satisfaction;
- evidence requirements for specific achievements;
- minimum collective progress thresholds;
- policy references governing closure semantics.

Criteria define what complete means in civic terms.

They are not ad hoc administrator discretion in Version 1.

## Required Milestone

A **Required Milestone** is a Milestone that must be satisfied for Completion Assessment to succeed.

Required Milestones:

- are defined by policy or approved implementation structure;
- appear in Completion Indicators and Community-facing progress presentation;
- remain distinct from Optional Milestones;
- contribute to derived Completion — not manual closure alone.

Required milestones express non-optional civic expectations.

They are not personal assignments.

## Optional Milestone

An **Optional Milestone** is a Milestone that informs orientation and progress context but is not required for Completion Assessment to succeed.

Optional Milestones:

- may still be recorded and celebrated as Achievements;
- must not block Completion when required criteria are satisfied;
- appear with clear non-blocking semantics in progress presentation.

Optional milestones add context.

They do not create hidden mandatory work.

## Future Coordination Space

**Future Coordination Space** is an intentionally deferred experience concept for richer coordination among participants during implementation.

Future Coordination Space is **not part of Version 1**.

It may eventually support structured collaboration patterns beyond collective progress recording.

Version 1 must not partially implement Coordination Space semantics through hidden task or messaging features.

The term exists to name future scope without importing it prematurely.

## Humanity Assistant

The **Humanity Assistant** is the platform's contextual guidance layer for civic participation across Capability 02.

Within Implementation, the Humanity Assistant:

- explains stage meaning and journey position after Implementation Commitment;
- clarifies what recorded achievements and derived progress mean;
- supports understanding before recommendation;
- may align with Next Meaningful Action when present.

Humanity Assistant principles:

- never replaces judgment;
- explains before recommending;
- reduces complexity, not responsibility;
- is contextual;
- is transparent.

The Humanity Assistant is not a coordinator, decision maker or achievement recorder.

## Policy Reference

A **Policy Reference** is a read-only link from Implementation to applicable Frozen Policy or approved policy context governing milestones, evidence rules, visibility or completion semantics.

Policy Reference:

- does not transfer policy ownership to Implementation;
- does not permit in-place mutation of frozen policy content;
- supports Completion Criteria and visibility interpretation;
- preserves aggregate independence.

Implementation reads policy context.

It does not own policy lifecycle.

## Implementation Readiness

**Implementation Readiness** is a **derived value** from the **Implementation Commitment** stage representing whether declared community capacity satisfied Frozen Policy thresholds for progression toward Implementation.

Within Epic 06 language:

- Implementation Readiness is an **eligibility input** from Stage 6 — not redefined by Implementation;
- Implementation may reference readiness context read-only when explaining why execution began;
- Implementation must not manually override or recompute Commitment-stage readiness as execution progress;
- readiness meant preparedness to proceed — not re-approval of collective direction.

```
Community Capacity (Commitment stage)

+

Frozen Policy

↓

Implementation Readiness (Commitment stage — input to Implementation eligibility)
```

Implementation records what is done after eligible progression.

Readiness records whether the community was prepared to begin.

## Derived values

**Collective Progress**, **Completion Assessment**, **Completion**, **Progress Indicators** and **Completion Indicators** are **derived values**.

They:

- are computed from recorded Achievements, Milestone state, Implementation Updates and applicable Completion Criteria;
- are read-only at API, store, workspace and public projection layers;
- must never become manually editable source of truth;
- must never appear as lifecycle states interchangeable with **Implementation Status**.

Recorded Achievements and accountable Implementation Updates remain authoritative source of truth over summaries.

## Evidence relationship

```
Achievement

↔
```

Evidence supports Achievements.

```
↔

Implementation Updates and Achievements (accountable recording)
```

# Anti-Terms

The following pairs must **never** be used as synonyms in Epic 06 language.

| Correct term | Do not substitute | Reason |
|--------------|-------------------|--------|
| **Achievement** | Task, Ticket, To-do, Work item | Achievements record collective accomplishment; tasks imply assignment and personal execution tracking |
| **Implementation** | Project Management, Program Management, Portfolio | Implementation records collective civic execution progress — not enterprise PM methodology |
| **Milestone** | Personal Assignment, Individual Deliverable | Milestones are collective civic checkpoints — not jobs assigned to a person |
| **Completion** | Manual Approval, Administrator sign-off | Completion is derived from criteria satisfaction — not discretionary override |
| **Progress** | Activity, Engagement, Session count | Progress reflects recorded collective accomplishment — not platform activity volume |
| **Evidence** | Opinion, Commentary, Endorsement | Evidence substantiates accomplishment factually — not persuasive belief |
| **Implementation** | Employment, HR record, Payroll | Implementation tracks civic execution — not employer–employee administration |
| **Coordinator** | Manager, Supervisor, Boss | Coordinators facilitate accountable recording within policy — not command hierarchy by default |
| **Assistant** | Decision Maker, Approver, Recorder | Assistants explain — they do not decide, approve or record accountable acts on behalf of participants |
| **Implementation Update** | Chat message, Comment thread | Updates are accountable civic records — not informal conversation |
| **Progress Report** | Performance review, Ranking report | Reports summarize collective state — not personal evaluation |
| **Implementation Readiness** | Completion, Go-live approval | Readiness belongs to Commitment stage preparedness; Completion belongs to Implementation execution closure |
| **Implementation Commitment** | Implementation | Commitment declares preparedness; Implementation records execution |
| **Milestone** | Deadline, Schedule slot | Milestones mark collective civic checkpoints — not calendar scheduling in Version 1 |
| **Public Implementation** | Implementation Workspace | Separate audiences, separate surfaces, separate DTOs |
| **Implementation Workspace** | Task board, Kanban, Backlog | Workspace orients and records collective progress — not personal task management |
| **Future Coordination Space** | Hidden task feature in Version 1 | Deferred concept must not enter scope through synonym drift |
| **Community Observer** | Operator, Moderator | Observers read by default — accountable roles require explicit policy |
| **Required Milestone** | Optional Milestone | Required milestones gate Completion Assessment; optional milestones do not |
| **Collective Progress** | Individual productivity score | Progress is community-level derivation — not personal metrics |
| **Completion Criteria** | Project plan, Roadmap | Criteria define civic closure conditions — not execution scheduling artifacts |

When in doubt, use the term defined in this document — not the nearest familiar product synonym.

# Future Evolution

The following concepts are **intentionally excluded from Version 1**.

They may appear in future Architecture Review scope.

They must not silently enter Epic 06 Version 1 through implementation convenience or synonym drift.

## Coordination Space

Structured coordination environments beyond collective progress recording.

## Task Assignment

Personal or role-based work assignment, ownership transfer or execution delegation inside the platform.

## Calendar

Shared or personal scheduling, deadline management and time-block coordination.

## Scheduling

Automated or manual assignment of people to times, shifts or execution slots.

## Messaging

Chat, threads, direct messages or notification-driven coordination replacing accountable Implementation Updates.

## Volunteer Coordination

Operational volunteer roster management, shift signup or workforce dispatch.

## External Tool Integration

Embedded project management, document collaboration or third-party execution systems owned by external aggregates.

---

Version 1 records **collective implementation progress only**.

Future extensions must preserve:

- aggregate independence;
- derived progress and completion discipline;
- operational/public separation;
- Human Leadership;
- calm civic language.

Deferred concepts must not increase present complexity.

# Final Principles

The following platform principles govern Epic 06 — Implementation language and architecture.

## The platform records collective achievements, not individual work logs

Implementation language centers community accomplishment, Milestones and Evidence — not personal task tracking or productivity surveillance.

## Progress is derived

Collective Progress and Progress Indicators are computed from recorded truth.

They are never manually edited as authority.

## Completion is derived

Completion and Completion Assessment are computed from Completion Criteria satisfaction.

They are never manual approval substitutes.

## Implementation follows Commitment

Implementation begins only after eligible pipeline progression.

Implementation Commitment readiness is input — not redefined or overridden by execution recording.

## Implementation does not replace external project management tools

The platform provides civic execution transparency.

It does not absorb enterprise PM, scheduling or messaging domains in Version 1.

## Assistant never replaces judgment

Humanity Assistant explains and orients.

Accountable Achievement recording and Implementation Updates remain human commands through canonical operational surfaces.

## Future expansion must not increase present complexity

Version 1 scope stays narrow.

Deferred concepts remain named but unimplemented until governance approves expansion.

Every architectural document, implementation guide and code artifact for Epic 06 — Implementation — must use the terminology defined in this document.

Implementation exists to answer one civic question with clarity and dignity:

**"What is being done?"**

Its language must preserve the distinction between preparedness, collective accomplishment and completion — so that transparency remains public, responsibility remains human and the Participation Pipeline remains trustworthy from commitment through impact.
