# Purpose

This document defines the ubiquitous language for the **Implementation Commitment** domain.

Its purpose is to establish one consistent vocabulary for architects, engineers, designers and future contributors.

Implementation Commitment is Stage 6 of the Humanity Union Participation Pipeline.

It follows approved collective direction and public endorsement.

It identifies who is prepared to help turn approved civic direction into responsible action.

This document defines language only.

It does not define implementation.

---

# Core Definition

## Implementation Commitment

An **Implementation Commitment** is the civic process through which registered participants declare preparedness to contribute to implementation of an approved direction.

An Implementation Commitment:

- follows eligible prior pipeline stages;
- collects declared contributions from participants;
- aggregates community capacity;
- derives implementation readiness;
- prepares a community for the Implementation stage.

An Implementation Commitment is not task management.

It is not employment.

It is the bridge between public support and responsible execution.

---

# Core Terms

## Contribution

A **Contribution**, in the Implementation Commitment domain, is a participant's declared willingness and capacity to help implement an approved direction.

A Contribution:

- belongs to one participant within one Implementation Commitment context;
- expresses intent and capacity — not completed work;
- may include skills, availability, resources or professional capacity;
- becomes part of the civic record when recorded under policy.

Contribution in this domain must not be confused with Collaborative Analysis Contribution, which belongs to understanding — not implementation preparedness.

## Contribution Profile

A **Contribution Profile** is the participant-owned summary of what a person is prepared to offer toward implementation.

A Contribution Profile may include:

- skills and experience relevant to community needs;
- availability windows;
- preferred contribution types;
- regional or organizational context where applicable.

The profile supports informed matching.

It does not assign work automatically.

## Contribution Item

A **Contribution Item** is one recorded declaration within a Contribution Profile for a specific Implementation Commitment.

Each Contribution Item:

- has a contribution type;
- expresses capacity within stated availability;
- carries a commitment status;
- remains immutable once recorded unless policy explicitly permits revision.

Contribution Items are the atomic unit of declared preparedness.

## Contribution Type

**Contribution Type** classifies the nature of a declared contribution.

Version 1 examples include:

- Volunteer contribution;
- Professional contribution;
- Resource contribution.

Contribution Type guides capability matching and community need alignment.

It does not determine employment relationship or compensation.

## Contribution Capacity

**Contribution Capacity** is the measurable preparedness a participant declares for a specific contribution type within stated availability.

Capacity is expressed in domain terms — hours, scope, skill level, resource quantity or equivalent policy-defined measures.

Capacity is declared by the participant.

It is not inferred or assigned by the platform on their behalf.

## Community Capacity

**Community Capacity** is the aggregate preparedness of all participants who have recorded contributions toward an Implementation Commitment.

Community Capacity:

- is derived from recorded Contribution Items;
- reflects declared skills, availability and resources;
- supports capability matching against community needs;
- is never manually edited as a substitute for recorded declarations.

Community Capacity describes what the community is prepared to offer.

It does not describe what has already been executed.

## Implementation Readiness

**Implementation Readiness** is a **derived value** representing whether declared community capacity satisfies frozen implementation policy for progression toward Implementation.

```
Community Capacity

+

Frozen Policy

↓

Implementation Readiness
```

Implementation Readiness:

- is computed — never manually entered;
- reflects policy thresholds and recorded capacity;
- may change when contributions are recorded or policy state changes;
- does not approve or reject the underlying collective decision.

Readiness means preparedness to proceed — not authorization to bypass governance.

## Implementation Policy

**Implementation Policy** defines the rules governing Implementation Commitment participation.

Implementation Policy may include:

- who may declare contributions;
- eligible contribution types;
- visibility rules;
- revision or withdrawal rules where permitted;
- readiness thresholds;
- coordinator responsibilities;
- progression requirements toward Implementation.

Implementation Policy follows the Living Policy Lifecycle.

It becomes enforceable only at Approved or Frozen stages as defined by policy governance.

## Living Policy

**Living Policy** is any participation policy that has not yet reached the **Frozen** stage.

Living Policy may evolve through deliberate review.

It must not be treated as permanently binding civic law until approved and frozen for its context.

Living Policy stages:

Suggested → Initial → Refined → Approved → Frozen → Satisfied

## Frozen Policy

**Frozen Policy** is implementation policy locked for a defined civic context or lifecycle window.

Frozen Policy:

- provides stable rules for eligibility, matching and readiness derivation;
- may change only through Architecture Review, Engineering Decision or approved policy version update;
- is a required input to Implementation Readiness calculation.

Frozen does not mean eternal.

It means intentionally stable until governance authorizes change.

## Policy Assistant

The **Policy Assistant** is a contextual guidance surface focused on **policy comprehension**.

The Policy Assistant:

- explains what a Living Policy or Frozen Policy means in plain civic language;
- clarifies eligibility, thresholds and participant obligations;
- helps participants understand readiness requirements before declaring contributions;
- operates within Humanity Assistant principles.

The Policy Assistant does not create, approve or freeze policy.

It does not record contributions on behalf of participants.

## Humanity Assistant

The **Humanity Assistant** is the platform's contextual guidance layer for civic participation across Capability 02.

Within Implementation Commitment, the Humanity Assistant:

- explains stage meaning and journey position;
- clarifies what declaring a contribution means;
- supports understanding before recommendation;
- may align with Next Meaningful Action and Participation Navigator when present.

Humanity Assistant principles:

- never replaces judgment;
- explains before recommending;
- reduces complexity, not responsibility;
- is contextual;
- is transparent.

The Humanity Assistant is not a coordinator, decision maker or commitment recorder.

## Readiness Threshold

A **Readiness Threshold** is a Frozen Policy requirement that declared Community Capacity must satisfy for Implementation Readiness to be reached.

Examples:

- minimum volunteer count for a skill category;
- required professional contribution type present;
- minimum aggregate availability within a time window;
- required resource contribution recorded.

Thresholds are policy-defined.

They are evaluated against derived capacity — not against manual readiness flags.

## Availability

**Availability** is the time, duration or window during which a participant declares they may contribute.

Availability:

- is declared by the participant;
- is associated with Contribution Items;
- supports realistic matching;
- does not guarantee assignment or execution.

Availability describes preparedness — not completed labor.

## Commitment Status

**Commitment Status** describes the lifecycle state of a recorded Contribution Item or aggregate commitment collection.

Commitment Status belongs to Implementation Commitment domain language.

It must not reuse Petition lifecycle states or Collective Decision statuses as synonyms.

Examples of commitment-oriented meaning (final states defined in domain model):

- declared;
- active;
- withdrawn where policy permits;
- satisfied when policy conditions are met.

Status describes the commitment record — not implementation task progress.

## Participant

A **Participant** is a registered Humanity Union member acting within operational participation surfaces.

In Implementation Commitment, a Participant may:

- declare contributions;
- maintain a Contribution Profile;
- review their recorded commitments;
- receive Contribution Recognition after accountable declarations.

Participant is the registered civic actor.

It is not interchangeable with Volunteer.

## Coordinator

A **Coordinator** is a participant or steward role authorized by Implementation Policy to facilitate commitment collection and matching.

A Coordinator:

- may help align community needs with declared capacity;
- may support orderly progression toward Implementation;
- does not replace participant judgment in declaring contributions;
- does not unilaterally assign implementation work.

Coordination is facilitation — not command authority unless explicitly defined by approved policy outside this document.

## Volunteer

A **Volunteer** is a participant who declares volunteer-type contribution capacity toward implementation.

Volunteer describes a **contribution type orientation** — unpaid civic preparedness to help.

Every Volunteer is a Participant.

Not every Participant is a Volunteer.

Volunteer must not be used as a generic label for all registered members.

## Professional Contribution

A **Professional Contribution** is a declared contribution type representing skilled professional capacity offered toward implementation within stated availability and policy rules.

Professional Contribution:

- expresses preparedness — not contracted employment;
- may require verification or policy eligibility;
- supports capability matching for specialized community needs.

It does not create an employer–employee relationship by declaration alone.

## Resource Contribution

A **Resource Contribution** is a declared contribution type representing material, financial, facility, equipment or organizational resources offered toward implementation.

Resource Contribution:

- expresses offered capacity — not transferred ownership by default;
- follows visibility and policy rules;
- supports readiness when thresholds require resource preparedness.

## Policy Satisfaction

**Policy Satisfaction** occurs when recorded Community Capacity and commitment state satisfy the requirements of applicable Frozen Policy for a defined readiness or progression condition.

Policy Satisfaction:

- is evaluated — not manually toggled;
- may correspond to Living Policy lifecycle **Satisfied** stage for a superseded policy generation;
- preserves auditable meaning of why readiness changed.

## Capability Matching

**Capability Matching** is the derived alignment process between **Community Need** and declared **Contribution Items**.

Capability Matching:

- compares declared skills, types, availability and resources against identified needs;
- supports coordinators and participants in understanding fit;
- does not auto-assign implementation tasks;
- preserves Human Leadership — people choose what to declare.

Matching informs preparedness.

It does not replace Implementation execution tracking.

## Next Meaningful Action

**Next Meaningful Action** is the single contextual recommendation presented to a participant after orientation and major state changes.

In Implementation Commitment, examples include:

- review community needs before declaring contribution;
- declare a contribution when eligible;
- review readiness progress after declaration;
- proceed toward Implementation workspace when readiness is satisfied and policy permits.

One recommendation only.

Calm language.

No urgency or gamification.

## Community Need

A **Community Need** is an identified implementation requirement derived from approved direction and subject context.

Community Need:

- describes what kind of help the community requires;
- may reference skills, roles, resources or availability windows;
- is not an assigned task list owned by Implementation Commitment execution;
- informs capability matching.

Community Need expresses requirement — not work completion.

## Participation Navigator

The **Participation Navigator** is the future platform service responsible for cross-stage journey guidance.

In Implementation Commitment context, the Navigator:

- identifies that the participant is in the Implementation Commitment stage;
- reads eligibility from aggregate state;
- recommends Next Meaningful Action consistently with pipeline position.

The Navigator is not an Aggregate.

It does not record contributions.

## Workspace

A **Workspace** is the operational environment where registered participants understand and act within one Aggregate stage.

The **Implementation Commitment Workspace** is the canonical surface for declaring contributions and reviewing readiness context.

Workspaces answer the five standard participant questions defined in the Participation Architecture Freeze.

## Operational View

The **Operational View** is the registered participant-facing operational surface backed by the operational API.

It includes:

- Contribution Profile and declaration surfaces;
- personal commitment status;
- derived readiness presentation;
- Contribution Recognition;
- Next Meaningful Action;
- journey context.

Operational View is authoritative for participant commitment state.

## Public View

The **Public View** is the read-only public projection surface for societal transparency.

It includes:

- aggregate readiness summary in public terms;
- community capacity indicators appropriate for public display;
- community needs summary where policy permits;
- participation entry guidance.

Public View must not expose individual participant identity beyond approved public visibility rules.

Public View is not the commitment declaration surface.

```

↓

```

```
→

Next Meaningful Action

→

```

## Pipeline position

```
Initiative

↓

Collaborative Analysis

↓

Collective Decision

↓

Petition

↓

Implementation Commitment

↓

Implementation

↓

Impact
```

Implementation Commitment references earlier stages by identifier and approved snapshot.

It does not mutate Initiative, Collaborative Analysis, Collective Decision or Petition state.

## Role relationships

```
Participant

├── may declare → Contribution Item

├── may hold → Contribution Profile

└── may act as → Volunteer (contribution type)

Coordinator

└── facilitates → Capability Matching (does not replace Participant declaration)

Humanity Assistant / Policy Assistant

└── explain → never record commitments on behalf of participants
```

# Anti-Terms

The following pairs must **never** be used as synonyms in Epic 05 language.

| Correct term | Do not substitute | Reason |
|--------------|-------------------|--------|
| **Implementation Commitment** | Support, Signature, Endorsement | Support belongs to Petition; commitment declares preparedness to help implement |
| **Support** | Commitment | Petition support does not imply implementation preparedness |
| **Volunteer** | Participant | Participant is the registered actor; Volunteer is one contribution type orientation |
| **Participant** | Volunteer | Not all participants are volunteers; not all contributions are volunteer-type |
| **Commitment** | Task, Assignment, Job | Commitment declares preparedness; tasks belong to Implementation execution |
| **Task** | Commitment, Contribution | Tasks are execution artifacts — not declaration of preparedness |
| **Implementation** | Implementation Commitment | Implementation Commitment prepares who may help; Implementation executes work |
| **Implementation Commitment** | Implementation | Commitment stage precedes execution stage |
| **Readiness** | Approval, Authorization | Readiness is derived preparedness; it does not re-approve the collective decision |
| **Approval** | Readiness | Collective Decision approval precedes this stage; readiness is not a second vote |
| **Contribution** | Employment, Contract, Hire | Declaring contribution is civic preparedness — not hiring |
| **Employment** | Professional Contribution | Professional contribution expresses offered skill capacity — not employment status |
| **Assistant** | Coordinator, Decision Maker, Steward of record | Assistants explain; they do not decide or record accountable acts |
| **Policy Assistant** | Humanity Assistant (as identical scope) | Policy Assistant is policy-focused; Humanity Assistant is platform-wide guidance |
| **Policy** | Project Plan, Roadmap, Backlog | Policy governs rules and thresholds — not execution scheduling |
| **Project Plan** | Implementation Policy | Plans belong to Implementation planning surfaces — not policy domain |
| **Community Capacity** | Headcount, Staffing level | Capacity is derived from declarations — not HR metrics |
| **Implementation Readiness** | Go-live flag, Manual checkbox | Readiness is computed — never manually overridden |
| **Capability Matching** | Auto-assignment | Matching aligns need and capacity — assignment belongs to human coordination |
| **Community Need** | Ticket, Issue, Task | Need describes requirement — not work item ownership |
| **Operational View** | Public View | Separate audiences, separate DTOs, separate surfaces |
| **Public View** | Operational View | Public transparency must not expose private commitment detail by default |
| **Contribution Recognition** | Praise, Ranking, Badge | Recognition confirms action — not personal worth |
| **Living Policy** | Frozen Policy | Living policy is evolvable; frozen policy is intentionally stable |
| **Frozen Policy** | Draft policy | Frozen requires explicit approval and lock semantics |
| **Collaborative Analysis Contribution** | Implementation Commitment Contribution | Analysis contributions build understanding; commitment contributions declare implementation preparedness |
| **Signature** | Commitment | Signatures endorse approved direction; commitments declare help toward implementation |
| **Petition Outcome** | Implementation Readiness | Petition outcome summarizes endorsement; readiness summarizes implementation preparedness |

When in doubt, use the term defined in this document — not the nearest familiar product synonym.

# Final Principle

Every architectural document, implementation guide and code artifact for Epic 05 — Implementation Commitment — must use the terminology defined in this document.

Implementation Commitment exists to answer one civic question with clarity and dignity:

**"Who is prepared to help?"**

Its language must preserve the distinction between endorsement, preparedness and execution — so that responsibility remains human, transparency remains public and the Participation Pipeline remains trustworthy from decision through action.
